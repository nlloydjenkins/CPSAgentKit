# CPSAgentKit вЂ” VS Code Extension Spec

## Purpose

A VS Code extension that turns GitHub Copilot into a Copilot Studio expert. It scaffolds CPS agent projects, injects curated platform knowledge into the workspace, and writes a `.github/copilot-instructions.md` that makes GHCP deeply aware of CPS constraints, multi-agent patterns, and best practices. The developer works with GHCP normally вЂ” it just happens to know everything about building CPS agents properly.

## Problem

Developers building Copilot Studio agents default to single-chatbot patterns because multi-agent orchestration with tool use is a complexity cliff. The platform can do far more than most people build. The hard part isn't the portal вЂ” it's the architecture decisions, the prompt engineering across multiple agents, and knowing the platform constraints that aren't well-documented.

## Who uses this

Developers and power users who are building CPS agents for business users. They're comfortable in VS Code and use the Copilot Studio VS Code extension. They are not necessarily CPS experts вЂ” that's what this tool provides.

## What it does

### Extension commands

- **Initialise CPS Project** (`cpsAgentKit.init`) вЂ” scaffolds the project folder structure into the current workspace. Creates the spec template, architecture template, knowledge folder, and writes the `copilot-instructions.md`. If a CPS agent project already exists in the workspace (from the CPS extension), it initialises around it non-destructively.
- **Sync Knowledge** (`cpsAgentKit.syncKnowledge`) вЂ” pulls the latest CPS platform knowledge from the central knowledge repo (a GitHub repo). Overwrites the local knowledge folder and regenerates the `copilot-instructions.md` to reflect any updates. Can be triggered manually or runs on workspace open if the project is already initialised.
- **Open Spec** (`cpsAgentKit.openSpec`) вЂ” opens the `requirements/spec.md` file for editing. Convenience command.

### The copilot-instructions.md

Generated by the extension by combining:

1. A static workflow section that defines the phases (Define в†’ Architect в†’ Build в†’ Test)
2. The contents of the knowledge folder, inlined or referenced
3. Awareness of the current project state (does requirements/spec.md exist? requirements/architecture.md? what agent folders exist? are there docs in requirements/docs/?)

The instructions tell GHCP:

- **On first interaction:** ask the developer what they need, read all documents in `requirements/docs/` for domain context, create `requirements/spec.md` with purpose, what it does, what it doesn't, what success looks like. Lightweight вЂ” 30-50 lines.
- **After spec exists:** read the spec, knowledge, and all documents in `requirements/docs/`, propose architecture in `requirements/architecture.md`. How many agents, what each does, what tools/connectors needed, how they relate. Be opinionated.
- **During build:** generate prompts, topic structures, knowledge chunk layouts. When something needs manual portal creation, say so explicitly with the settings to use. Track state in `requirements/architecture.md`.
- **When test output is pasted:** evaluate against `requirements/spec.md`. Diagnose what went wrong. Suggest specific changes.
- **Cross-agent consistency:** when a child agent's scope changes, flag that the parent's routing instructions need updating.

### Extension settings

- `cpsAgentKit.knowledgeRepoUrl` вЂ” URL of the central knowledge GitHub repo (default: TBD)
- `cpsAgentKit.syncOnOpen` вЂ” whether to sync knowledge on workspace open (default: true)
- `cpsAgentKit.knowledgeRepoBranch` вЂ” branch to pull from (default: main)

## Integration with the Copilot Studio VS Code Extension

CPSAgentKit is designed to work alongside the official Copilot Studio VS Code extension (pre-release). Understanding how that extension works is critical to our design.

### How the CPS extension works

The CPS extension provides a full local authoring experience for CPS agents:

1. Developer signs in to Copilot Studio via the extension
2. Clones an agent вЂ” this downloads the agent's editable components (topics, knowledge sources, actions, triggers) as **YAML files** into a local folder
3. Edits are made directly to these YAML files in VS Code (with IntelliSense support)
4. Changes are synced back via three commands:
   - **Preview changes** вЂ” diff local vs remote without modifying anything
   - **Get changes** вЂ” pull latest from CPS into local workspace
   - **Apply changes** вЂ” push local edits to the CPS environment (live editing, immediate effect)

### CPS extension command IDs

These commands are exposed programmatically and can be invoked via `vscode.commands.executeCommand()`:

| Command ID                                      | Description                           |
| ----------------------------------------------- | ------------------------------------- |
| `microsoft-copilot-studio.applyChanges`         | Push local edits to CPS environment   |
| `microsoft-copilot-studio.getChanges`           | Pull latest from CPS to local         |
| `microsoft-copilot-studio.previewChanges`       | Diff local vs remote                  |
| `microsoft-copilot-studio.cloneAgent`           | Clone an agent to local workspace     |
| `microsoft-copilot-studio.reattachAgent`        | Reattach to a previously cloned agent |
| `microsoft-copilot-studio.refreshAgentTreeView` | Refresh the agents list               |
| `microsoft-copilot-studio.signIn`               | Sign in to Copilot Studio             |
| `microsoft-copilot-studio.switchAccount`        | Switch CPS account                    |
| `microsoft-copilot-studio.reportIssue`          | Report an issue with session info     |
| `microsoft-copilot-studio.sessionInfo`          | Show current session info             |

### What this means for CPSAgentKit

**GHCP already has full visibility of the agent.** Because the CPS extension clones agent components as YAML files in the workspace, GHCP can see and edit topics, triggers, actions, knowledge configs, and instructions through normal file context. Our extension does not need to provide any special file access.

**Our extension scaffolds around the cloned agent.** When initialising, if a CPS agent already exists in the workspace (YAML files from the CPS extension), CPSAgentKit creates its files (spec, architecture, knowledge, instructions) alongside without modifying the agent files.

**The sync model is live editing.** Apply Changes pushes directly to the environment вЂ” no staging, no branching. The spec and architecture docs we create are the only version control and intent-tracking layer.

**Automated deploy is possible in v2.** The `microsoft-copilot-studio.applyChanges` command can be invoked programmatically, enabling a future deploy в†’ test в†’ evaluate loop without manual palette interaction.

## Technical implementation

### Extension structure

- **Language:** TypeScript
- **Extension API version:** latest stable VS Code Extension API
- **Activation:** on command, or when a `.cpsagentkit` folder is detected in the workspace
- **UI:** minimal вЂ” commands in the command palette, a status bar item showing sync status. No webview panel in v1.

### Knowledge sync mechanism

- Central knowledge repo is a GitHub repo
- Config stored in `.cpsagentkit/config.json` with the repo URL and last sync timestamp
- Sync pulls the `/knowledge` folder from the repo via GitHub API (raw content) and writes to `.cpsagentkit/knowledge/`
- After sync, regenerates `copilot-instructions.md` from the template + knowledge content
- Sync runs on "Sync Knowledge" command and optionally on workspace open

### copilot-instructions.md generation

Assembled from:

1. The `copilot-instructions-template.md` in templates (static workflow + GHCP role definition)
2. The contents of each knowledge file, formatted as sections
3. Project state awareness вЂ” checks for existence of requirements/spec.md, requirements/architecture.md, requirements/docs/, and agent folders

The file is regenerated on sync and should not be manually edited.

## Workflow phases

### Phase 1: Define

Developer tells GHCP what they need. GHCP reads any documents in `requirements/docs/`, asks clarifying questions, and creates `requirements/spec.md`:

- Purpose (one paragraph)
- What the agent should do (bullet points)
- What the agent should not do (bullet points)
- What success looks like (concrete examples)
- Domain knowledge references (which docs the agent needs)

### Phase 2: Architect

GHCP reads the spec, CPS knowledge, and all documents in `requirements/docs/`, then creates `requirements/architecture.md`:

- Number of agents and the role of each
- Parent-child relationships and routing logic
- Tools/connectors needed and which agent owns each
- Knowledge sources and how they map to agents
- What needs manual portal creation (listed explicitly)

### Phase 3: Build

Iterative loop:

1. GHCP generates/modifies agent prompts, topic definitions, knowledge structures
2. GHCP identifies what needs manual portal action
3. Developer does the portal work, applies changes via CPS extension
4. Developer tests in CPS portal test pane
5. Developer pastes output into GHCP
6. GHCP evaluates against spec, suggests changes
7. Repeat

### Phase 4: Test (manual in v1)

Developer tests using scenarios from the spec's success criteria. Pastes outputs into GHCP for evaluation.

## What it does not do

- Does not replace the CPS VS Code extension
- Does not interact with the CPS portal or Direct Line API (v1)
- Does not create agents programmatically
- Does not build a custom GHCP Chat Participant
- Does not hand-hold вЂ” assumes VS Code and GHCP competency

## Success criteria

- A developer with no CPS multi-agent experience can describe a business problem and get a working multi-agent architecture with correct prompts
- Generated configurations respect CPS platform constraints
- When test output is pasted back, GHCP gives actionable diagnosis
- Knowledge updates propagate on next sync without reinstalling

## Future (not v1)

- **Automated deploy + test loop:** call `microsoft-copilot-studio.applyChanges` programmatically, then run Direct Line test, evaluate against spec, suggest fixes вЂ” all without leaving VS Code
- **Direct Line integration:** automated testing against the spec using the Bot Framework Direct Line API
- **Architecture visualisation:** webview panel showing agent graph with parent-child relationships and tool assignments
- **Handover package generation:** solution zip with documentation for business users
- **Chat Participant (`@cps`):** if the instructions file approach hits context limits
- **Template library:** pre-built architectures for common patterns (IT triage, HR onboarding, compliance checking)
- **CPS extension detection:** auto-detect when a CPS agent is cloned into the workspace and offer to initialise CPSAgentKit around it
