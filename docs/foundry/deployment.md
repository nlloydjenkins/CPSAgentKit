# Deployment

## HTTP Server Pattern

Foundry agents are deployed as HTTP servers. The Agent Framework provides a hosting adapter that wraps an agent or workflow as a RESTful endpoint.

### Packages

```bash
pip install azure-ai-agentserver-agentframework==1.0.0b16 azure-ai-agentserver-core==1.0.0b16
```

### Basic Pattern

```python
from azure.ai.agentserver.agentframework import from_agent_framework

# For a single agent
await from_agent_framework(agent).run_async()

# For a workflow-as-agent
agent = workflow.as_agent()
await from_agent_framework(agent).run_async()
```

### HTTP Request Handling

The starter executor must accept `list[Message]` input:

```python
@handler
async def handle_request(self, messages: list[Message], ctx: WorkflowContext[...]) -> None:
    ...
```

To return output via HTTP response, use `ctx.yield_output()` with `AgentResponseUpdate`:

```python
from agent_framework import AgentResponseUpdate, Content

await ctx.yield_output(
    AgentResponseUpdate(
        contents=[Content("text", text=response.text)],
        role="assistant",
        author_name=self.id,
    )
)
```

### Environment Variables

```
FOUNDRY_PROJECT_ENDPOINT=<project-endpoint>
FOUNDRY_MODEL_DEPLOYMENT_NAME=<model-deployment-name>
```

Use `load_dotenv(override=False)` so deployed environment variables take precedence.

## Foundry Hosted Agents

Deploy via the AI Toolkit and Microsoft Foundry VS Code extensions:

1. Ensure the app is wrapped as an HTTP server (above)
2. Run VS Code command: **Microsoft Foundry: Deploy Workflow** (`azure-ai-foundry.commandPalette.deployWorkflow`)

The extension handles containerisation (Docker), ACR push, and Foundry hosted agent registration.

Alternatively from the editor title bar when a YAML workflow file is open:

- `azure-ai-foundry.editorTitle.deployYamlWorkflow` â€” deploy YAML-based workflow
- `azure-ai-foundry.editorTitle.deployPromptAgent` â€” deploy prompt agent

## Local Testing

Before deploying, test locally using the AI Toolkit:

1. **Run Agent** â€” `azure-ai-foundry.commandPalette.runAgent` starts the agent locally
2. **Agent Inspector** â€” `ai-mlstudio.openTestTool` opens interactive debugging UI (port must match `agentdev --port` argument, default 8088)
3. **Container Playground** â€” `azure-ai-foundry.commandPalette.runInContainerAgentPlayground` runs in a containerised environment

## Debug Configuration

Include VS Code launch configurations for local development:

- `agentdev` CLI for quick local testing
- AI Toolkit Agent Inspector for interactive debugging

These are generated by the `aitk-add_agent_debug` tool during scaffolding.

See [ai-toolkit-extension.md](ai-toolkit-extension.md) for the full command reference.

## Key Difference from CPS

CPS uses **Apply Changes** (`microsoft-copilot-studio.applyChanges`) to push YAML config directly to the CPS environment â€” live editing with no staging. Foundry uses a standard software deployment model: code â†’ Docker â†’ ACR â†’ hosted agent, triggered via `azure-ai-foundry.commandPalette.deployWorkflow`. This gives you version control, CI/CD pipelines, and rollback by default.

| CPS Workflow Step | Foundry Equivalent                                                 |
| ----------------- | ------------------------------------------------------------------ |
| Clone Agent       | Create New Agent (`ai-mlstudio.createNewAgent`)                    |
| Edit YAML         | Edit Python code                                                   |
| Preview Changes   | Run Agent locally (`azure-ai-foundry.commandPalette.runAgent`)     |
| Apply Changes     | Deploy Workflow (`azure-ai-foundry.commandPalette.deployWorkflow`) |
